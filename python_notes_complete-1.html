Collections

When to Use Which? — Decision Guide

Use LIST when:
You need an ordered collection that changes frequently.
Example: To-do items, cart items, log entries.
Use TUPLE when:
Data should not change after creation. Slightly faster. Good for returning multiple values from a function.
Example: RGB color (255, 128, 0), DB row, config values.
Use SET when:
You want unique values only or need fast membership check (is X in set?).
Example: Unique tags, removing duplicates from list, set operations (union, difference).
Use DICTIONARY when:
You need to label your data with names (keys).
Example: User profile, JSON-like data, counting word frequency.
# Practical: Remove duplicates from list using set
names = ["Vinu", "Ravi", "Vinu", "Arun", "Ravi"]
unique_names = list(set(names))
print(unique_names)  # ['Ravi', 'Arun', 'Vinu']

# Practical: Count word frequency using dict
sentence = "the cat sat on the mat the cat"
freq = {}
for word in sentence.split():
    freq[word] = freq.get(word, 0) + 1
print(freq)  # {'the': 3, 'cat': 2, 'sat': 1, 'on': 1, 'mat': 1}

# Practical: Check if user already visited (set lookup is O(1))
visited = {"Mysuru", "Bangalore", "Chennai"}
if "Bangalore" in visited:
    print("Already visited!")
🧠 Quick Rule: Need labels? → Dict. Need unique? → Set. Need fixed? → Tuple. Need flexible ordered? → List.

class Student:
    school = "NIE Mysuru"          # Class variable — shared by all

    def __init__(self, name, marks):
        self.name = name           # Instance variable — unique to each object
        self.marks = marks

    def greet(self):               # Instance method
        return f"Hi, I am {self.name} from {Student.school}"

    def is_pass(self):
        return self.marks >= 40

    @classmethod
    def get_school(cls):           # Class method
        return cls.school

    @staticmethod
    def grade(marks):              # Static method — no self or cls needed
        return "A" if marks >= 75 else "B" if marks >= 60 else "C"

# Creating objects (instances)
s1 = Student("Vinu", 92)
s2 = Student("Ravi", 55)

print(s1.greet())            # Hi, I am Vinu from NIE Mysuru
print(s1.is_pass())          # True
print(Student.get_school())  # NIE Mysuru
print(Student.grade(92))     # A
Why self? When you call s1.greet(), Python automatically passes s1 as the first argument. self is just the name we give to this argument (by convention). It always refers to this specific object.

Section 3 — OOP Four Pillars

OOP — Pillar 1

Inheritance — "Borrow from Parent"

Definition: Inheritance allows a child class to reuse properties and methods of a parent class. The child can also add new features or override (change) parent methods.

Why use it: Avoids code repetition. You write common features once in the parent, and all children get it automatically.

🧠 Memory Trick: Think of a real family 👨‍👩‍👧.
Parent = Person (has name, age)
Child = Student (has everything Person has + roll_no, marks)
Just like you inherit your father's last name — Student inherits Person's features.
TypeDescriptionExample
SingleOne parent, one childAnimal → Dog
Multi-levelGrandparent → Parent → ChildAnimal → Dog → Labrador
MultipleChild inherits from 2 parentsFlyingFish(Bird, Fish)
HierarchicalOne parent, many childrenAnimal → Cat, Dog, Bird
 
# PARENT class
class Person:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def introduce(self):
        return f"I am {self.name}, age {self.age}"

# CHILD class — inherits Person
class Student(Person):
    def __init__(self, name, age, roll_no, marks):
        super().__init__(name, age)   # calls parent __init__
        self.roll_no = roll_no
        self.marks = marks

    def study_info(self):
        return f"Roll: {self.roll_no}, Marks: {self.marks}"

    # OVERRIDE parent method
    def introduce(self):
        return f"I am {self.name}, a student with roll {self.roll_no}"

# Multi-level inheritance
class Topper(Student):
    def award(self):
        return f"{self.name} is the class topper!"

# Usage
s1 = Student("Vinu", 24, "CS101", 92)
print(s1.introduce())      # Uses Student's overridden version
print(s1.study_info())     # Own method

t1 = Topper("Ravi", 23, "CS102", 98, )
# TypeError above — Topper needs same args as Student, fix:
t1 = Topper("Ravi", 23, "CS102", 98)
print(t1.introduce())      # inherited from Student
print(t1.award())          # own method
super() — calls the parent class method. Always use super().__init__() in child's __init__ to properly initialize parent attributes.

Polymorphism — "Same Name, Different Behavior"

Definition: Polymorphism means many forms. The same method name works differently in different classes.

Why use it: Write one common interface, but each class behaves differently. Reduces complexity.

🧠 Memory Trick: Think of the word "speak" 🗣️
Dog speaks → Bark 🐕
Cat speaks → Meow 🐈
Human speaks → Hello 👋
Same action speak(), completely different result. That is polymorphism.
Method Overriding — child redefines parent method 
class Animal:
    def speak(self):
        return "Some sound"

class Dog(Animal):
    def speak(self):       # override
        return "Bark!"

class Cat(Animal):
    def speak(self):       # override
        return "Meow!"

# Polymorphic behavior
animals = [Dog(), Cat(), Animal()]
for a in animals:
    print(a.speak())
# Bark!
# Meow!
# Some sound
Duck Typing — Python doesn't check type, just behavior 
# No inheritance needed!
# If it has .area(), we treat it as a shape
class Circle:

        self.r = r
    def area(self):

Key point: In Python, polymorphism happens naturally. If the method name is the same, Python just calls it — no special keyword needed like in Java/C++.
OOP — Pillar 3

Encapsulation — "Data Hiding + Controlled Access"

Definition: Encapsulation means bundling data (attributes) and methods together inside a class, and restricting direct access to sensitive data. You control access through getter/setter methods.

Why use it: Protects data from accidental changes. Controls who can read/write what.

🧠 Memory Trick: Think of an ATM Machine 🏧
You cannot directly touch the money inside (private). You must use the ATM interface (deposit/withdraw methods) to access it.
Your bank balance = private attribute. Withdraw button = setter method.
Access LevelSyntaxMeaning
Publicself.nameAccessible from anywhere
Protectedself._salary (one _)Convention: "don't touch from outside" (but technically accessible)
Privateself.__balance (two __)Truly hidden. Name-mangled by Python.
 
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner          # public
        self._account_no = "XYZ"   # protected (convention)
        self.__balance = balance    # private (truly hidden)

    # GETTER — read private data
    def get_balance(self):
        return self.__balance

    # SETTER — controlled write
    def deposit(self, amount):
        if amount > 0:
            self.__balance += amount
            return f"Deposited ₹{amount}. New balance: ₹{self.__balance}"
        return "Invalid amount"

    def withdraw(self, amount):
        if 0 < amount <= self.__balance:
            self.__balance -= amount
            return f"Withdrawn ₹{amount}"
        return "Insufficient balance"

acc = BankAccount("Vinu", 5000)

# Direct access to private — FAILS
# print(acc.__balance)        # AttributeError!

# Correct way through method
print(acc.get_balance())      # 5000
print(acc.deposit(1000))      # Deposited ₹1000. New balance: ₹6000
print(acc.withdraw(2000))     # Withdrawn ₹2000
Python's name mangling: __balance becomes _BankAccount__balance internally. So acc._BankAccount__balance works technically, but you should never do this. It's bad practice.
OOP — Pillar 4

Abstraction — "Show Only What's Needed"

Definition: Abstraction means hiding the internal implementation and showing only the essential features. You define what a method should do, not how. Achieved using abstract classes.

Why use it: Forces all child classes to implement required methods. Creates a contract/interface.

🧠 Memory Trick: Think of a Car Dashboard 🚗
You see: Speedometer, Fuel gauge, Gear. You do NOT see: Engine pistons, fuel injection logic.
The dashboard shows only what you need. Everything else is hidden (abstracted away). 
from abc import ABC, abstractmethod   # ABC = Abstract Base Class

class Shape(ABC):                     # Abstract class — cannot be instantiated
    @abstractmethod
    def area(self):                   # Must be implemented by every child
        pass

    @abstractmethod
    def perimeter(self):
        pass

    def describe(self):               # Concrete method — shared by all
        return f"Area: {self.area()}, Perimeter: {self.perimeter()}"

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def area(self):                   # must implement
        return round(3.14 * self.radius ** 2, 2)

    def perimeter(self):
        return round(2 * 3.14 * self.radius, 2)

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w, self.h = w, h

    def area(self):
        return self.w * self.h

    def perimeter(self):
        return 2 * (self.w + self.h)

# s = Shape()      # ERROR! Cannot instantiate abstract class
c = Circle(7)
r = Rectangle(4, 6)

print(c.describe())   # Area: 153.86, Perimeter: 43.96
print(r.describe())   # Area: 24, Perimeter: 20
Key Rule: If a child class inherits from an abstract class and does NOT implement all abstract methods, Python will throw a TypeError. This is the "contract" — every child MUST implement all abstract methods.

# ─── 1. WRITE to a file (creates if not exists, overwrites if exists)
with open("notes.txt", "w") as f:
    f.write("Hello Vinu!\n")
    f.write("Python file handling is easy.\n")

# ─── 2. READ entire file
with open("notes.txt", "r") as f:
    content = f.read()
    print(content)

# ─── 3. READ line by line
with open("notes.txt", "r") as f:
    for line in f:
        print(line.strip())

# ─── 4. READ all lines into a list
with open("notes.txt", "r") as f:
    lines = f.readlines()
    print(lines)  # ['Hello Vinu!\n', 'Python file handling is easy.\n']

# ─── 5. APPEND to file (does NOT delete existing content)
with open("notes.txt", "a") as f:
    f.write("This line is added at the end.\n")

# ─── 6. CREATE file (fails if already exists)
try:
    with open("newfile.txt", "x") as f:
        f.write("Brand new file!")
except FileExistsError:
    print("File already exists!")

# ─── 7. DELETE a file safely
import os
try:
    os.remove("notes.txt")
    print("File deleted.")
except FileNotFoundError:
    print("File not found.")
Always use with open(). It automatically closes the file even if an error occurs. Without it, you must call f.close() manually — and if an error happens before that, the file stays open (resource leak).
File Handling

File Handling with Exception Handling

Real-world file operations always need error handling. Files can be missing, corrupted, or have permission issues.

# Complete safe file read
def read_file(filename):
    try:
        with open(filename, "r") as f:
            return f.read()
    except FileNotFoundError:
        return f"Error: '{filename}' not found."
    except PermissionError:
        return "Error: No permission to read this file."
    except Exception as e:
        return f"Unexpected error: {e}"
    finally:
        print("read_file() completed.")   # always runs

print(read_file("data.txt"))

# ─── Write + Read same file (r+ mode)
with open("log.txt", "w") as f:
    f.write("Log entry 1\nLog entry 2\n")

with open("log.txt", "r+") as f:
    content = f.read()
    print("Before:", content)
    f.write("Log entry 3\n")    # appended at end of current position

# ─── Working with CSV-like data
students = [
    {"name": "Vinu", "marks": 92},
    {"name": "Ravi", "marks": 78},
]

with open("students.txt", "w") as f:
    for s in students:
        f.write(f"{s['name']},{s['marks']}\n")

# Read back
with open("students.txt", "r") as f:
    for line in f:
        name, marks = line.strip().split(",")
        print(f"{name} → {marks}")
🧠 File Handling Formula:
with open(file, mode) as f: → inside block use f.read() / f.write()
Wrap in try/except for safety. Use finally for cleanup.

  17. What are dunder methods? Special methods with __ like __str__, __len__, __eq__.
  18. What is shallow copy? New outer object but nested objects are shared.
  19. What is deep copy? Fully independent copy including all nested objects.
  20. What is JSON? Text format for data exchange. Python uses json.dumps/loads.
  21. What is regex? Pattern-based text matching using re module.
  22. == vs is? == checks value equality; is checks object identity (memory).

Theory Questions — Click to Reveal Answers

Q1: What is the difference between List, Tuple, Set, and Dictionary?

Answer: List = ordered, mutable, allows duplicates []. Tuple = ordered, immutable, allows duplicates (). Set = unordered, mutable, no duplicates {}. Dictionary = ordered (Python 3.7+), key-value pairs, keys must be unique {}. Use list for changeable sequences, tuple for fixed data, set for unique values, dict for labeled data.

Q2: What are the 4 pillars of OOP?

Answer: (1) Inheritance — child reuses parent code. (2) Polymorphism — same method, different behavior. (3) Encapsulation — hide data, control access. (4) Abstraction — show only interface, hide implementation. Memory trick: IPEA.

Q3: What is the difference between Encapsulation and Abstraction?

Answer: Encapsulation = bundling data + methods and hiding data using private attributes. Focus: HOW data is protected. Abstraction = hiding internal logic and showing only what's necessary. Focus: WHAT the object does. Encapsulation uses __private, getters/setters. Abstraction uses ABC, @abstractmethod.

Q4: What file modes are available in Python?

Answer: r (read), w (write — overwrites), a (append — adds to end), x (create — fails if exists), r+ (read+write), w+ (write+read), b (binary mode — used as rb, wb). Always use with open() for safe auto-close.

Q5: What is the difference between a class variable and an instance variable?

Answer: Class variable is shared by all objects — defined at class level (e.g., school = "NIE"). Instance variable is unique to each object — defined in __init__ using self (e.g., self.name). If class variable changes, all objects see it. If instance variable changes, only that object is affected.

Q6: What is pip and why do we use a virtual environment?

Answer: pip is Python's package manager — installs libraries from PyPI. Virtual environment is an isolated Python environment per project. It prevents library version conflicts between projects. Use: python -m venv myenv, then activate it.

Q7: What is super() and when do you use it?

Answer: super() calls the parent class method from inside a child class. Most commonly used in __init__ of child class: super().__init__(name, age) — this ensures the parent's initialization code runs before adding child-specific attributes.

Q8: What is the difference between @classmethod and @staticmethod?

Answer: @classmethod receives cls (the class itself) as first argument — used to access/modify class variables. @staticmethod receives no automatic argument — it's just a utility function that logically belongs to the class but doesn't need the object or class. Example: Student.grade(90) as static, Student.get_school() as classmethod.
